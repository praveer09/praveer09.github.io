# Phase 3 — Wire up redirects

> **Goal:** generate, from the single source of truth (each post's `redirectFrom` +
> `migration/url-inventory.csv`), the two redirect artifacts that guarantee **no old URL
> 404s**:
> 1. `public/_redirects` — Cloudflare-native **301s** for the `praveergupta.in` site.
> 2. **Static redirect stub pages** for the old `praveer09.github.io` deep links (so they keep
>    working even though GitHub Pages can't do server-side 301s).

Reasoning: `../research/greenfield-blueprint.md` §0 (strategy A: redirect), §0a (`_redirects`
mechanics, the GitHub Pages 301 caveat), §4 (why Cloudflare's native `_redirects` matters).

---

## 3.1 — Why two artifacts (read this once)

| Old URL host | Who serves it after cutover | Redirect mechanism |
|---|---|---|
| `praveergupta.in/<slug>-<hex>` (Medium paths) | **Cloudflare Pages** (same host as new site) | `_redirects` → **301** (best for SEO) |
| `praveergupta.in/feed` | Cloudflare Pages | `_redirects` → 301 to `/rss.xml` |
| `praveer09.github.io/<category>/<date>/<slug>/` | **GitHub Pages** (different host) | **Static stub page** (`<link rel=canonical>` + `<meta http-equiv=refresh>`) — GitHub Pages can't issue real 301s |

Both are generated from the same data, so they can never drift. `_redirects` also gets the
github.io-style paths added to it (harmless, and covers any edge case where they're requested
on the canonical host).

---

## 3.2 — Generator 1: `_redirects` (Cloudflare 301s)

Create `scripts/gen-redirects.mjs`. It must run as a **plain Node script** (it's a prebuild
step, so it can't use Astro's `getCollection`/`astro:content` — those only resolve inside
Astro's runtime). Instead it parses post frontmatter directly with `gray-matter`.

> 🤖 **Prompt:** "Write `scripts/gen-redirects.mjs` (Node, ESM, **no `astro:content`
> imports**). It should: glob `src/content/blog/**/index.{md,mdx}`, parse each file's
> frontmatter with `gray-matter`, derive the slug from the post's folder name, and for each
> `redirectFrom` entry emit `<old_path>  /blog/<slug>/  301`; then read
> `migration/url-inventory.csv` and emit every non-`post` row as `<old_path>  <new_path>  301`
> (feed, about, home if needed, `/sitemap.xml` → `/sitemap-index.xml`, tags decision);
> de-duplicate; write to `public/_redirects`. Only put the splat `*` at the **end** of a
> source path, and do **not** emit any `404` lines. Add `gray-matter` and `fast-glob` as dev
> deps. Add a `pnpm run gen:redirects` script, then run it and show me `public/_redirects`."

**Sanity checks the script must enforce** (`../research/greenfield-blueprint.md` §0a):
- Free-tier limit is **2,000 static + 100 dynamic** redirects — you have ~60, so fine.
- Splat (`*`) only valid at the end of the source; prefer the **exact** harvested path.
- No `/* ... 404` line — the default `404.html` handles misses.

Example output:
```
# generated by scripts/gen-redirects.mjs — do not edit by hand
/practical-guide-to-java-stream-api-7aadc02908f7      /blog/practical-guide-to-java-stream-api/  301
/p/7aadc02908f7                                       /blog/practical-guide-to-java-stream-api/  301
/technology/2018/11/24/practical-guide-to-java-stream-api/  /blog/practical-guide-to-java-stream-api/  301
/feed                                                 /rss.xml                                   301
/feed.xml                                             /rss.xml                                   301
```

## 3.3 — Generator 2: `github.io` redirect stub pages

GitHub Pages can't issue server-side 301s, so for **every** old github.io URL (not just posts)
we emit a tiny HTML page at the old path that (a) declares the new URL as canonical and (b)
meta-refreshes to it. These are written into `public/` so they land in `dist/` and are served
by **GitHub Pages** (Phase 5 deploys `dist` there).

> 🤖 **Prompt:** "Write `scripts/gen-redirect-stubs.mjs` (Node, ESM; parse the CSV directly,
> **no `astro:content`**). For **every** row in `migration/url-inventory.csv` where
> `old_host = praveer09.github.io` **and** the `old_path` differs from its `new_path` — this
> covers post deep links (`/<category>/<date>/<slug>/`), `/about-me/` → `/about/`,
> `/feed.xml` → `/rss.xml`, `/sitemap.xml` → `/sitemap-index.xml`, and any `tags` rows whose
> path changes — write `public/<old_path>/index.html` (or `public/<old_path>` if it already
> ends in a file extension like `.xml`) containing a minimal HTML doc with
> `<link rel=\"canonical\" href=\"https://praveergupta.in<new_path>\">`,
> `<meta http-equiv=\"refresh\" content=\"0; url=https://praveergupta.in<new_path>\">`, a
> `<meta name=\"robots\" content=\"noindex\">`, and a visible fallback link. Skip rows where
> `old_path == new_path` (e.g. the home page in Option A, which is served directly). Add
> `pnpm run gen:stubs` and include it in the prebuild step."

> Note on `.xml` paths: a meta-refresh only works in an HTML response, so the stub for
> `/sitemap.xml` / `/feed.xml` is an HTML page served at that path. Crawlers that hit the old
> XML path get redirected to the new one; this is best-effort for the github.io host only (on
> the canonical Cloudflare host these same paths get a real 301 from `_redirects`).

> Why `noindex` on stubs: you don't want the stub pages themselves indexed — the `canonical`
> tells search engines the real page is on `praveergupta.in`.

---

## 3.4 — Absolute canonical on **every** page (not just posts)

Because the **same `dist/` is also served on `praveer09.github.io`** (Phase 5), every page —
home, about, tag pages, *and* posts — is duplicated on the github.io host. To make that
SEO-safe, **every** page must emit an **absolute** canonical pointing at its `praveergupta.in`
URL, so search engines consolidate authority on the canonical domain regardless of which host
served the HTML.

> 🤖 **Prompt:** "Add a shared SEO head partial used by the base layout so **every** page emits
> an **absolute** `<link rel=canonical href=\"https://praveergupta.in<pathname>\">` (built from
> the `site` config + the page's own path — posts resolve to `/blog/<slug>/`). Add OpenGraph/
> Twitter tags and, on posts, JSON-LD `BlogPosting`. The canonical must be identical whether
> the page is served from Cloudflare or GitHub Pages."

> This is why absolute (not relative) canonicals matter: a relative canonical on the github.io
> duplicate would point at `praveer09.github.io`, defeating the consolidation.

## 3.5 — Wire generators into the build

```jsonc
// package.json
"scripts": {
  "gen:redirects": "node scripts/gen-redirects.mjs",
  "gen:stubs": "node scripts/gen-redirect-stubs.mjs",
  "prebuild": "node scripts/gen-redirects.mjs && node scripts/gen-redirect-stubs.mjs",
  "build": "astro build"
}
```

> Note: Astro runs `prebuild` automatically before `build` (npm/pnpm lifecycle), so the
> redirect artifacts are always regenerated and in sync with content.

### Findings / decisions recorded during 3.x

- **Cloudflare static-asset precedence:** on Cloudflare Pages, static assets take precedence over
  `_redirects`. So github.io-style stub paths soft-redirect (canonical + meta-refresh) on the
  canonical host instead of hard-301 — acceptable, since the canonical-host inbound links are the
  Medium **flat** paths, which have no stub and therefore get real 301s.
- **`.xml` paths are NOT stubbed** (`/feed.xml`, `/sitemap.xml`): an HTML meta-refresh is useless to
  XML clients and a stub would shadow the real `_redirects` 301 on Cloudflare. They are handled by
  `_redirects` only.
- **`/tags/` route added** (`src/pages/tags/index.astro` + `[tag].astro`) so `/tagged/*` → `/tags/`
  resolves; added explicit CSV row for the base `/tagged` (the `/tagged/*` splat does not match it).
- **5 parked Medium-only posts** (empty `new_slug`/`new_path`) intentionally 404 until recreated from
  the Medium export. Listed in the `_redirects` header comment and allowlisted in
  `scripts/verify-redirects.mjs`.
- **Deploy requirement:** generated artifacts are gitignored, so every deploy MUST run
  `pnpm run build` (NOT `astro build` directly) so the prebuild step emits `_redirects` + stubs.
  Phase 4 (Cloudflare build command) and Phase 5 (GitHub Pages workflow) must use `pnpm run build`.
